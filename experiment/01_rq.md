RQ chính thức
------------

RQ1 (Branch Coverage của GPT-4)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Các test case do GPT-4 tự động sinh ra có đạt mức độ bao phủ nhánh (branch coverage) tối thiểu 75% trên các hàm Java và Python có độ phức tạp trung bình không?

RQ2 (So sánh với Baselines)
~~~~~~~~~~~~~~~~~~~~~~~~~~~
Chất lượng kiểm thử (đo qua Mutation Score) của bộ test do GPT-4 sinh ra có cao hơn mang ý nghĩa thống kê so với công cụ truyền thống (Randoop) không?

RQ3 (Ảnh hưởng của độ phức tạp)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Độ phức tạp Cyclomatic Complexity (CC) ảnh hưởng như thế nào đến chất lượng test case (branch coverage và mutation score) do GPT-4 sinh ra?

----------------------------------------------------

PICO Framework
--------------

P (Population/Problem)
~~~~~~~~~~~~~~~~~~~~~~
- Dataset: 50 hàm mã nguồn (25 hàm Java + 25 hàm Python) từ các dự án mã nguồn mở trên GitHub.
- Đặc điểm: Các hàm độc lập có độ phức tạp Cyclomatic Complexity ở mức trung bình (từ 5 đến 15, đo bằng Lizard).
- Nguồn: Các GitHub repositories của các thư viện phổ biến.

I (Intervention)
~~~~~~~~~~~~~~~~
- LLM: GPT-4 (phiên bản gpt-4o).
- Parameters: Cấu hình API với temperature=0.2 để đảm bảo độ ổn định cao.
- Strategy: Zero-shot prompting chứa chữ ký hàm (method signature), mã nguồn và docstring/JavaDoc.

C (Comparison)
~~~~~~~~~~~~~~
- Loại so sánh: So sánh với công cụ truyền thống:
  1. System Baseline: Các bộ test case được sinh tự động bởi công cụ truyền thống Randoop (áp dụng cho các hàm Java).

O (Outcome)
~~~~~~~~~~~
- Primary metrics: Branch Coverage (%) đo bằng JaCoCo (cho Java) và Coverage.py (cho Python).
- Secondary metrics: Mutation Score (%) đo bằng PIT Mutation Testing (cho Java) và pytest-mutagen (cho Python) để đánh giá năng lực phát hiện lỗi thực tế.
- Cách đo: 
  1) Chạy JaCoCo/Coverage.py để lấy tỷ lệ phần trăm bao phủ nhánh.
  2) Chạy PIT/pytest-mutagen để tính toán Mutation Score của từng bộ test.
  3) Thực hiện các phép kiểm định thống kê phi tham số (Wilcoxon signed-rank, Mann-Whitney U, Spearman) qua thư viện scipy.stats để kiểm chứng p-value (p < 0.05).

----------------------------------------------------

RQ đầy đủ (sau khi tổng hợp PICO)
---------------------------------

RQ tổng quát:
"Đối với 50 hàm Java và Python có độ phức tạp cyclomatic complexity trung bình từ 5 đến 15 (P), bộ unit test do GPT-4 sinh ra (I) so với test case sinh bởi Randoop (C) có đạt branch coverage >= 75% và mutation score >= 65% mang ý nghĩa thống kê hay không, khi đo bằng JaCoCo/PIT và Coverage.py/pytest-mutagen (O)?"

----------------------------------------------------

Sub-RQs
-------

RQ1.1: GPT-4 có đạt branch coverage trung vị >= 75% trên Java methods không?
[H0: median(branch_coverage_GPT4_Java) < 75% | H1: median(branch_coverage_GPT4_Java) >= 75% | Test: Wilcoxon signed-rank]

RQ1.2: GPT-4 có đạt branch coverage trung vị >= 75% trên Python methods không?
[H0: median(branch_coverage_GPT4_Python) < 75% | H1: median(branch_coverage_GPT4_Python) >= 75% | Test: Wilcoxon signed-rank]

RQ2: GPT-4 có đạt mutation score cao hơn Randoop trên Java không?
[H0: median(mutation_GPT4_Java) <= median(mutation_Randoop_Java) | H1: median(mutation_GPT4_Java) > median(mutation_Randoop_Java) | Test: Mann-Whitney U]

RQ3.1: Cyclomatic complexity có tương quan nghịch với branch coverage của test do GPT-4 sinh ra không?
[H0: rho >= 0 | H1: rho < 0 | Test: Spearman's rank correlation]

Tips khi viết RQ
----------------

Tip 1: RQ phải testable
- Cần có ngưỡng so sánh định lượng rõ ràng (ví dụ: > 80% coverage) thay vì các tính từ mơ hồ như "hiệu quả", "tốt hơn".

Tip 2: RQ phải khả thi (Feasible)
- Kích thước mẫu 50 hàm (25 Java + 25 Python) là khả thi để đo đạc thủ công và chạy kiểm thử đột biến kỹ lưỡng trong thời gian làm tiểu luận.

Tip 3: RQ phải giải quyết khoảng trống nghiên cứu (GAP)
- RQ phải trực tiếp trả lời cho các khoảng trống đã tìm thấy ở Phần 1 (ví dụ: thiếu đánh giá trên độ phức tạp CC và so sánh kiểm định giả thuyết thống kê).
